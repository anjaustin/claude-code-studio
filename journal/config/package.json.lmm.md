# LMM — package.json

## RAW

This is a tiny manifest for what HANDOFF says is a sprawling 7-phase desktop app: 9 runtime deps, 12 devDeps, 5 scripts, a single postinstall hook. The runtime dependency set tells the real story — `node-pty` (native), `@xterm/xterm` + `addon-fit` (terminal rendering), `electron-store@^11` (config persistence), `@octokit/rest@^21` (GitHub), `systeminformation` (resource polling), `react` and `react-dom` at v19, and `electron-squirrel-startup` for Windows installer plumbing. There is no testing framework, no linter, no formatter, no type-check script, no build verification script. The `main` field points to `.vite/build/index.js` which is generated by `electron-forge start` / `package` via the Vite plugin — so the entrypoint only exists after a build. `private: true` is correct because there is no publishConfig and no intent to publish to npm. The version is pinned at 1.0.0 from day one, which feels aspirational and pre-conventional (no 0.x phase). Electron is hard-pinned to `42.2.0` (no caret) — the only pinned dep — implying someone got burned by an Electron auto-bump and locked it. TypeScript is `^6.0.3`, which is far ahead of mainstream (TS 5.x is current as of late 2025); same with React 19, Vite 8, Node types 25. Open questions: (1) Why is `electron-store@^11` (the ESM-only major) chosen given the main process is bundled by Vite — is the `external: ['electron-store']` declaration in vite.main.config.ts enough, or is dynamic import required? (2) Where does the postinstall fail safely if VS Build Tools aren't installed? (3) Why no `electron-rebuild` listed as a devDep when HANDOFF tells humans to run `npx electron-rebuild` manually?

## NODES

1. **No CI/test/lint scripts** — only Forge wrappers. Quality gates are entirely manual.
2. **Pinned electron 42.2.0** vs **caret-ranged everything else** — explicit fragility signal around native rebuild compatibility.
3. **electron-store @^11** is ESM-only; main bundle is CJS-flavored (`.vite/build/index.js`). Externalization in vite.main.config.ts is the bridge — but a `require('electron-store')` in CJS output cannot load an ESM module synchronously.
4. **postinstall runs unconditionally** — patches node-pty even on machines without VS Build Tools. The patch script no-ops cleanly if node-pty is absent, but it does not detect "node-pty present but rebuild impossible."
5. **No electron-rebuild devDep** — HANDOFF tells the human to `npx electron-rebuild` on every fresh clone, fetching it ad-hoc.
6. **`main: ".vite/build/index.js"`** — only exists post-build, so `electron .` without a prior build silently fails. Forge's `start` masks this.
7. **React 19 + TypeScript 6** — bleeding edge for an app whose primary value is terminal emulation; risk-reward tilted toward "I want the newest stuff."
8. **No `engines` field** — Node version requirement is undeclared despite native deps needing matching toolchains.
9. **Author email matches the user's email in memory** — single-developer project, no team conventions to satisfy.
10. **License MIT** with `private: true` — contradictory if literally interpreted but harmless; signals "open source eventually" intent.

**Tension A**: Tight Electron pin vs. caret on everything else — partial defensive posture, leaves transitive native breakage exposed.
**Tension B**: Single postinstall script as the linchpin between checkout and working binary, but no `engines`/`os` constraints to fence what postinstall must contend with.

## REFLECT

Core insight: **this manifest is a single-author bench manifest, not a shipped product manifest** — the absence of test/lint/typecheck scripts and `engines` reveals that the author has accepted "it builds on my machine" as the contract.

Tension A resolved: the Electron pin exists because Electron bumps break native rebuilds (node-pty); the rest don't because they're pure JS. Asymmetric pinning is correct, just undocumented.

Tension B resolved: postinstall is permissive (no-ops on missing node-pty, doesn't validate toolchain) precisely because requiring an `engines`/`os` block would block CI/codespaces. The cost is silent failures on the human side that surface during `electron-rebuild`, not `npm install`.

Hidden assumptions: (1) every dev machine is Windows 11 with VS Build Tools 2022 + Win10 SDK; (2) the developer will read HANDOFF before running `npm start`; (3) electron-store v11 ESM works as an external because the Vite main bundle is actually emitted as ESM (likely true given `"module": "ESNext"` in tsconfig — but not enforced anywhere here); (4) version 1.0.0 is a label, not a release contract.

## SYNTHESIZE

What this should become:
- Add `engines.node` (and ideally `os: ["win32"]` since node-pty patches assume Windows).
- Add `electron-rebuild` as a devDep so `npx` doesn't re-resolve every run.
- Add `scripts.typecheck` (`tsc --noEmit`) and a `scripts.rebuild-pty` wrapper so HANDOFF's manual steps become callable.
- Either lock electron-store to v10 (CJS) or explicitly note in a comment that v11 is loaded via `await import()` in main.
- Drop version to 0.1.0 until Phase 7 ships, or commit to semver discipline.

Actionable items:
1. Add `"typecheck": "tsc --noEmit"` and run it in pre-commit / CI.
2. Move from `npx electron-rebuild` to a local devDep + script.
3. Audit electron-store usage in src/main to confirm ESM compatibility (separate journal).
4. Add `os` field to gate non-Windows `npm install` from running the winpty patches.

Risks if untouched: the next contributor (or the author on a new laptop) will fail at `npm start` with a cryptic native-module error, and the postinstall log buried mid-`npm install` output will be the only clue.
